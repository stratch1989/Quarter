# Analytics Report: Welcome Popup / Onboarding

> Автор: Senior UX-аналитик | Дата: 2026-03-25
> Методология: эвристический анализ, cognitive walkthrough, GOMS-анализ

## 0. Резюме для занятых

**Рекомендация:** Вариант E (Setup Mode - нампад в фокусе) -- превращаем главный экран при первом запуске в двухшаговый onboarding, где пользователь вводит бюджет и дату прямо через существующий нампад, без единого нового экрана. 5 тапов, ~12 секунд, zero new UI components.

---

## 1. Текущее состояние (As-Is)

### 1.1 Что видит новый пользователь

При первом запуске:
- Дисплей `result` = "0" + текст "/день" (Bebas Neue, 70sp, белый)
- Дисплей ввода `value` = "0" (60sp, белый, правый нижний угол)
- Подсказка `day_limit` = "Введите дату и сумму в настройках" (серый #888888, ~14sp, малозаметная)
- Нампад 4x4 с кнопками del/undo/enter справа
- Панель эмодзи-категорий (HorizontalScrollView)
- Поле заметки (noteField), поля даты и времени
- Шестеренка настроек (36dp, правый верхний угол)
- Кнопка истории (36dp, левый верхний угол)
- Кнопка стрика, кнопка подписок

### 1.2 Текущий путь настройки (6+ действий через 2 открытия меню)

```
Шаг 1: Тап на шестеренку ......................... +1 тап
Шаг 2: В попапе тап "Укажите бюджет" ............ +1 тап
Шаг 3: Ввод числа на нампаде (допустим 30000) ... +5 тапов
Шаг 4: Тап Enter (сохраняет бюджет) ............. +1 тап
        [попап закрывается автоматически]
Шаг 5: Снова тап на шестеренку ................... +1 тап
Шаг 6: Тап "Дата" ................................ +1 тап
Шаг 7: Прокрутка DatePickerDialog к нужной дате .. +3-5 тапов
Шаг 8: Тап OK .................................... +1 тап
                                          Итого: ~14-16 тапов
```

**Время:** ~25-40 секунд (измерено когнитивно)

### 1.3 Проблемы текущего подхода

| # | Проблема | Тяжесть |
|---|----------|---------|
| 1 | Подсказка незаметна -- серый на черном, мелкий шрифт | Критическая |
| 2 | Нужно дважды открыть настройки (нет последовательного flow) | Высокая |
| 3 | Пользователь может начать тыкать нампад, ничего не произойдет (0 - 0 = 0) | Высокая |
| 4 | Нет обратной связи "что делать дальше" после ввода бюджета | Средняя |
| 5 | DatePickerDialog -- стандартный Android, разрыв стиля с темным UI | Средняя |
| 6 | Пользователь видит пустую историю, стрики, подписки -- перегрузка | Низкая |

### 1.4 Техническая архитектура, влияющая на решение

- Фрагменты загружаются в `R.id.place_holder` (FrameLayout поверх всего)
- `EveryDayQuestion` -- пример модального фрагмента с полупрозрачным фоном #B3000000
- `History` -- пример popup-фрагмента с анимацией alpha 0->1 за 200ms
- Settings -- PopupWindow (не фрагмент), взаимодействует с нампадом через `isBudgetInputMode`
- `dialog_bg.xml` -- dark card: #1c1c1e, radius 25dp
- `history_popup_bg.xml` -- dark card: #181818, border gradient #555555->#4c4740, radius 24dp
- `button_border7.xml` -- кнопки действий: прозрачный фон, белый border 2dp, radius 360dp
- DataModel: `money`, `dateFull`, `keyTodayLimit` -- три ключевых LiveData
- `saveBudgetInput()` -- уже умеет принимать бюджет через нампад

## 2. Глубокий анализ вариантов A-D

### 2.1 Вариант A: Wizard (два шага: бюджет -> дата)

**Описание:** Два полноэкранных фрагмента-шага. Первый: "Какой у вас бюджет?" + поле ввода. Второй: "До какого числа?" + календарь/пикер.

**Пользовательский путь:**
```
[Первый запуск]
  -> Экран 1: заголовок + поле ввода + кнопка "Далее"
     Тап на поле -> системная клавиатура -> ввод "30000" -> тап "Далее"
  -> Экран 2: заголовок + календарь/пикер + кнопка "Готово"
     Выбор даты -> тап "Готово"
  -> Главный экран с рассчитанным лимитом
```

**Эмоциональный отклик:**
- Шаг 1: "Ага, понятно, нужно ввести бюджет" -- ощущение контроля
- Переход 1->2: "Логично, теперь дату" -- ожидаемо
- Завершение: "Готово, быстро настроил" -- удовлетворение
- НО: "Зачем системная клавиатура, если тут есть нампад?" -- легкое раздражение

**Когнитивная нагрузка:** 2/5
- Два простых решения: число и дата
- Линейный путь, нет ветвлений

**Точки отказа:**
- Системная клавиатура перекрывает контент на маленьких экранах
- Пользователь может нажать Back на шаге 2 и потерять бюджет
- Нет возможности пропустить (а вдруг пользователь хочет сначала осмотреться)

**Технические риски:**
- Средние: нужно 2 новых фрагмента + 2 layout + навигация между ними
- Системная клавиатура может конфликтовать с layout
- Нужен собственный календарь для dark theme или кастомизация DatePickerDialog

**Accessibility:** 4/5
- Стандартные Android-компоненты (EditText, DatePicker)
- Хорошо работает с TalkBack
- Но: два экрана = больше навигации для пользователей screen reader

### 2.2 Вариант B: Полноэкранный фрагмент (свой нампад)

**Описание:** Один полноэкранный фрагмент со встроенным нампадом для ввода бюджета и кнопкой выбора даты.

**Пользовательский путь:**
```
[Первый запуск]
  -> Полноэкранный фрагмент:
     Сверху: "Ваш бюджет: _" (крупно)
     Снизу: встроенный нампад
     Ввод числа -> тап Enter -> появляется выбор даты
     Выбор даты -> тап "Готово"
  -> Главный экран
```

**Эмоциональный отклик:**
- Открытие: "О, нампад. Нужно ввести число" -- интуитивно
- Ввод числа: "Крупные кнопки, удобно" -- привычно
- Переход к дате: "А, еще дату надо" -- может удивить
- Завершение: "Круто, теперь я вижу лимит" -- удовлетворение

**Когнитивная нагрузка:** 2/5
- Один экран, два последовательных действия
- Нампад знаком из калькуляторов

**Точки отказа:**
- Нампад на onboarding-экране может смутить ("это калькулятор?")
- Переход от ввода числа к выбору даты на одном экране -- сложная анимация
- Нет контекста для числа (рублей? долларов? за какой период?)

**Технические риски:**
- Высокие: дублирование нампада (уже есть в activity_main)
- Или переиспользование нампада -- сложная архитектурная задача
- Нужен пикер даты внутри этого же фрагмента

**Accessibility:** 3/5
- Кастомный нампад может не поддерживать TalkBack корректно
- Нужна ручная разметка contentDescription для каждой кнопки

### 2.3 Вариант C: Setup Mode (главный экран в режиме настройки)

**Описание:** Главный экран показывается, но нампад работает в "режиме настройки" -- сначала вводишь бюджет, потом дату.

**Пользовательский путь:**
```
[Первый запуск]
  -> Главный экран, НО:
     - result показывает "Бюджет: _"
     - Все лишнее скрыто/затемнено (категории, заметка, история, стрик)
     - Нампад работает
  -> Ввод числа на нампаде -> тап Enter
  -> result показывает "До какого числа?"
  -> Появляется DatePicker (или кастомный выбор)
  -> Выбор даты -> тап "Готово"
  -> Все элементы появляются, экран полностью функционален
```

**Эмоциональный отклик:**
- Открытие: "Хм, что-то спрашивают. Нампад уже виден" -- умеренный интерес
- Ввод бюджета: "Просто набираю число. Удобно" -- привычно
- Переход к дате: "А, теперь дату" -- логично
- Появление полного UI: "Вау, всё ожило!" -- положительный сюрприз

**Когнитивная нагрузка:** 1.5/5
- Минимальный UI = минимум отвлечений
- Нампад уже знаком (та же сетка кнопок)
- Последовательный flow без переходов между экранами

**Точки отказа:**
- Непонятно что "бюджет" значит без контекста (на месяц? на год?)
- Тап Back может закрыть приложение (нет "предыдущего шага")
- Если пользователь случайно ввел число -- как исправить?

**Технические риски:**
- Средние: нужно управлять состоянием (isSetupMode) в MainActivity
- Скрытие/показ элементов уже реализовано (showNoteMode / hideNoteMode -- паттерн alpha animate)
- Переиспользование нампада -- без дублирования кода
- Пикер даты -- можно использовать существующий DatePickerDialog

**Accessibility:** 4/5
- Существующий нампад с уже настроенным contentDescription
- Системный DatePicker
- Меньше UI-элементов -- проще для screen reader

### 2.4 Вариант D: Попап-карточка с двумя полями

**Описание:** Модальная карточка (как EveryDayQuestion) с двумя полями -- бюджет и дата -- и кнопкой "Начать".

**Пользовательский путь:**
```
[Первый запуск]
  -> Главный экран + поверх:
     Карточка (dialog_bg style):
       "Добро пожаловать!"
       [Поле бюджета: _____]
       [Поле даты: dd.mm.yyyy]
       [Кнопка: Начать]
  -> Тап на поле бюджета -> системная клавиатура -> ввод
  -> Тап на поле даты -> DatePicker
  -> Тап "Начать"
  -> Карточка закрывается
```

**Эмоциональный отклик:**
- Открытие: "Форма для заполнения. Ладно" -- нейтрально
- Ввод бюджета: "Стандартное поле, ОК" -- привычно
- Выбор даты: "DatePicker, понятно" -- ожидаемо
- Закрытие: "Наконец-то, давайте к делу" -- нетерпение

**Когнитивная нагрузка:** 2.5/5
- Два поля + кнопка = три элемента для обработки
- Форма -- привычный паттерн, но вызывает "form fatigue"
- Два разных способа ввода (клавиатура + пикер) -- переключение контекста

**Точки отказа:**
- Системная клавиатура может перекрыть поле даты
- "Начать" неактивна пока оба поля не заполнены -- пользователь может не понять почему
- Карточка может быть слишком маленькой для длинных чисел

**Технические риски:**
- Низкие: один фрагмент, стандартные View-компоненты
- Стиль dialog_bg уже существует (#1c1c1e, radius 25dp)
- Но: системная клавиатура + карточка = проблемы с позиционированием

**Accessibility:** 4/5
- Стандартные EditText + DatePicker
- Хорошо работает с TalkBack
- Понятная форма

## 3. Новые варианты (E, F, G)

### 3.1 Вариант E: "Guided Numpad" -- пошаговый онбординг через существующий нампад

**Концепция:** Нампад уже есть на экране. При первом запуске он просто переключается в режим ввода бюджета (аналог isBudgetInputMode), но вместо попапа настроек -- крупный текст-подсказка прямо на дисплее. После ввода бюджета -- автоматически открывается DatePicker для даты.

**Пользовательский путь:**
```
[Первый запуск]
  -> Главный экран, НО:
     - result (70sp, Bebas Neue) показывает "_" (мигающий курсор)
     - day_limit (увеличен, bold) показывает "Укажите бюджет"
     - value скрыт
     - Остальные элементы затемнены (alpha 0.15, как в noteMode)
     - Нампад работает в полную силу

  Шаг 1: Ввод бюджета
     Пользователь набирает "30000" на нампаде .... 5 тапов
     result показывает "30 000" (форматировано)
     Тап Enter ...................................... 1 тап

  Шаг 2: Выбор даты
     day_limit меняется на "До какого числа?"
     result показывает "30 000 /день"
     Автоматически появляется DatePickerDialog ..... 0 тапов
     Пользователь крутит дату ........................ 2-3 тапа
     Тап OK .......................................... 1 тап

  -> Все элементы плавно появляются (alpha 0.15 -> 1.0)
  -> result показывает рассчитанный лимит "1000"
  -> Приложение готово к работе
                                          Итого: ~9-10 тапов
```

**Эмоциональный отклик:**
- Открытие: "Мигающий курсор -- надо что-то набрать. Написано 'Укажите бюджет'. Понятно!" -- мгновенное понимание
- Ввод числа: "Кнопки большие, число видно крупно. Классно" -- тактильное удовольствие + визуальная обратная связь
- Тап Enter: "Готово. А, теперь дату надо" -- плавный переход
- DatePicker: "Стандартно, быстро" -- привычно
- Появление UI: "О, вот и лимит! 1000 в день -- отлично" -- мгновенный результат
- Первая трата: "Теперь нампад работает для трат. Ясно!" -- осознание УТП

**Когнитивная нагрузка:** 1/5
- Один фокус внимания: мигающий курсор + текст-подсказка
- Нампад уже виден = 0 cognitive overhead
- Линейный flow без ветвлений
- Нет новых экранов, нет переключения контекста

**Точки отказа:**
- Пользователь может нажать Back -- нужно блокировать или показать "Вы уверены?"
- Может ввести 0 или слишком маленькое число -- нужна валидация
- Может закрыть DatePicker без выбора -- нужен fallback (открыть снова или предложить "позже")
- Если пользователь убьет приложение между шагом 1 и 2 -- бюджет сохранится, дата нет

**Технические риски: НИЗКИЕ**
- `isBudgetInputMode` уже существует и работает -- просто активируем его при первом запуске
- Затемнение элементов -- копия паттерна `showNoteMode()` / `hideNoteMode()`
- DatePickerDialog -- уже используется в settings
- Нужен флаг `isFirstSetup` в SharedPreferences
- Мигающий курсор -- `ObjectAnimator.ofFloat(view, "alpha", 1f, 0f)` с `repeatMode = REVERSE`
- **Нет новых фрагментов, нет новых layout-файлов**

**Accessibility:** 5/5
- Существующий нампад с contentDescription
- Системный DatePicker
- Минимум элементов -- screen reader зачитает только подсказку и кнопки

**Почему это лучше варианта C:**
Вариант C описывает общую концепцию "setup mode", но E конкретизирует: переиспользуем isBudgetInputMode, не создаем новый режим. Разница архитектурная -- E минимально инвазивен.

### 3.2 Вариант F: Bottom Sheet с двумя шагами

**Концепция:** При первом запуске снизу выезжает BottomSheet, перекрывая нампад. Два шага в одном sheet: бюджет (с собственным мини-нампадом или полем ввода) и дата (встроенный календарь).

**Пользовательский путь:**
```
[Первый запуск]
  -> Главный экран виден сверху
  -> Снизу выезжает BottomSheet:
     "Сколько у вас денег на этот период?"
     [Большое поле ввода: _____]
     [Кнопка: Далее ->]

  Шаг 1: Тап на поле -> системная клавиатура -> ввод "30000" -> тап "Далее"

  -> BottomSheet анимирует переход к шагу 2:
     "До какого числа?"
     [Мини-календарь или список дат]
     [Кнопка: Начать]

  Шаг 2: Выбор даты -> тап "Начать"

  -> BottomSheet сворачивается вниз
  -> Нампад появляется
```

**Эмоциональный отклик:**
- Появление: "О, что-то выехало. Интересно" -- внимание захвачено
- Ввод: "Стандартная форма, ОК" -- нейтрально
- Анимация перехода: "Красиво" -- положительно
- Закрытие: "Наконец нампад" -- облегчение (но и значит ожидание было негативным)

**Когнитивная нагрузка:** 2/5
- Привычный паттерн BottomSheet (Google Maps, Uber, и др.)
- Два шага, но в одном контейнере
- Системная клавиатура -- знакомо

**Точки отказа:**
- BottomSheet + системная клавиатура = война за пространство (классическая Android-проблема)
- Мини-календарь в sheet -- мало места, сложно пользоваться
- Свайп вниз закроет sheet -- пользователь потеряет прогресс
- Нампад скрыт за sheet -- пользователь не видит основное УТП приложения

**Технические риски: СРЕДНИЕ**
- BottomSheetDialogFragment -- стандартный компонент Material, но проект не использует Material Components
- Проект использует raw PopupWindow и кастомные фрагменты -- BottomSheet будет инородным
- Конфликт с нампадом (занимают одно место)
- Нужен кастомный календарь для dark theme

**Accessibility:** 3.5/5
- BottomSheet хорошо поддерживается TalkBack
- Но: системная клавиатура может перекрыть элементы
- Нужна правильная разметка шагов для screen reader

### 3.3 Вариант G: "Chat-like" пошаговый диалог

**Концепция:** Вдохновлено чат-ботами и приложением "Тяжеловато". Вместо формы -- диалог с пользователем. "Привет! Сколько денег осталось до зарплаты?" -> пользователь вводит число -> "А когда зарплата?" -> выбирает дату -> "Отлично! Ваш лимит: 1000/день".

**Пользовательский путь:**
```
[Первый запуск]
  -> Вместо главного экрана:
     Темный фон, сверху:
     💬 "Привет!"
     💬 "Сколько денег у вас есть?"
     Снизу: нампад (тот же самый)

  Шаг 1: Ввод "30000" -> тап Enter
     Появляется:
     🙂 "30 000"
     💬 "Отлично! До какого числа нужно растянуть?"

  Шаг 2: Появляется DatePicker или список дат
     Выбор -> тап OK
     Появляется:
     🙂 "30 апреля"
     💬 "Можно тратить 1000 в день!"

  -> Плавный переход к главному экрану (3 секунды)
```

**Эмоциональный отклик:**
- Открытие: "О, мне говорят привет! Мило" -- теплое первое впечатление
- Вопрос о деньгах: "Прямой вопрос, понятно что вводить" -- ясность
- Ответ-подтверждение: "Подтвердили мой ввод, хорошо" -- уверенность
- Результат: "1000 в день -- круто, понятно!" -- мгновенная ценность
- НО: "Это приложение для трат или чат-бот? Стиль не совпадает" -- диссонанс

**Когнитивная нагрузка:** 1.5/5
- Разговорный формат -- самый понятный для человека
- Один вопрос за раз = минимум нагрузки
- Но: отнимает больше времени из-за анимаций сообщений

**Точки отказа:**
- Анимации сообщений замедляют процесс (вау-эффект vs скорость)
- Стиль "чат" диссонирует с минималистичным калькуляторным UI
- Пользователь может подумать что это туториал который надо пролистать, а не интерактив
- Back button -- куда возвращаться?

**Технические риски: ВЫСОКИЕ**

## 5. Финальная рекомендация

### 5.1 Основной вариант: E (Guided Numpad)

**Почему:**

1. **Минимальная сложность реализации.** Механизм `isBudgetInputMode` уже работает. Паттерн затемнения `showNoteMode()`/`hideNoteMode()` уже реализован. DatePickerDialog уже интегрирован. Нужен только флаг `isFirstSetup` и ~60 строк кода.

2. **Консистентность 100%.** Пользователь с первой секунды видит тот же нампад, который будет использовать ежедневно. Нет "обучающего" экрана, который выглядит иначе чем приложение. Мышечная память формируется с первого взаимодействия.

3. **Подчеркивает УТП.** Ключевое преимущество Quarter -- трата записывается за 1 секунду через нампад. В варианте E пользователь знакомится с нампадом ещё до первой траты. В вариантах A, D, F нампад скрыт за формами.

4. **Минимальный риск отказа.** Нет нового UI, нет форм, нет переходов между экранами. Ввод числа на нампаде -- интуитивно для любого пользователя (опыт калькулятора).

5. **Отличная accessibility.** Существующие элементы с уже настроенными contentDescription. Screen reader: "Укажите бюджет. Кнопка 1. Кнопка 2..." -- всё стандартно.

### 5.2 Запасной вариант: C (Setup Mode -- общая концепция)

**Почему запасной:** Если по каким-то причинам переиспользование `isBudgetInputMode` окажется невозможным (например, конфликты с PopupWindow-логикой), вариант C дает свободу реализовать то же самое через отдельный `isSetupMode` флаг с чуть большим количеством кода.

### 5.3 Что категорически НЕ делать

1. **НЕ Chat-like (G).** Стилистически чужеродный для минималистичного калькуляторного UI. Высокая сложность реализации при сомнительном ROI. Создает ложное впечатление о характере приложения.

2. **НЕ полноэкранный фрагмент с дублированием нампада (B).** Дублирование кода нампада -- архитектурный анти-паттерн. Нампад в Quarter -- не просто 10 кнопок, а сложная система с bounce-анимацией, dimOverlay, режимами -- дублировать это недопустимо.

3. **НЕ BottomSheet (F).** Проект не использует Material Components. BottomSheet будет инородным элементом. Конфликт с нампадом за пространство экрана.

4. **НЕ добавлять Skip/Пропустить.** Если пользователь пропустит -- он увидит "0 /день" и бесполезное приложение. Настройка бюджета и даты -- необходимый минимум для работы. Пропуск = гарантированный отток.

### 6.4 Обработка edge cases

| Edge case | Поведение |
|-----------|-----------|
| **Back button на шаге 1 (ввод бюджета)** | Блокировать. Приложение закрывается через системный жест Home, но Back не должен уходить "назад" -- некуда. Аналог: EveryDayQuestion блокирует Back через OnBackPressedCallback. |
| **Back button на шаге 2 (DatePicker)** | DatePicker закрывается. Показать его снова через 500ms. Или: если пользователь закрыл DatePicker, показать кнопку "Выбрать дату" в day_limit (как fallback). |
| **Ввод 0 или пустого бюджета** | Enter игнорируется. Подсказка мигает красным на 300ms, возвращается к оранжевому. |
| **Ввод отрицательного числа** | Кнопка +/- заблокирована в setup mode (isSetupMode -> plusMinus disabled). |
| **Ввод слишком большого числа (>99999999)** | Ограничение как в isBudgetInputMode: `budgetInputValue.replace(".", "").length >= 8`. |
| **Поворот экрана** | onSaveInstanceState сохраняет isFirstSetup + budgetInputValue. Или: configChanges="orientation" в манифесте (уже может быть). |
| **Закрытие приложения между шагом 1 и 2** | При следующем запуске: если money > 0 но dateFull не задана -- показать только шаг 2 (DatePicker). Проверка: `howMany != 0.0 && numberOfDays <= 0`. |
| **DatePicker: пользователь выбрал сегодня** | Невозможно -- `picker.datePicker.minDate = tomorrow` (уже реализовано). |
| **DatePicker: отмена (Cancel)** | Показать снова через 500ms. Через 2 отмены -- оставить пользователя на экране с кнопкой "Выбрать дату" вместо day_limit. |
| **Системная клавиатура появляется** | Не должна -- noteField не получает фокус в setup mode. Можно добавить `binding.noteField.isFocusable = false` на время setup. |

### 6.5 Микроинтеракции

1. **Bounce кнопок нампада** -- без изменений (setupBounceAnimation). Работает в setup mode, создает тактильное ощущение "кнопки нажимаются".

2. **Обратная связь при вводе числа** -- result обновляется мгновенно, шрифт 70sp. Крупное число = визуальное подтверждение.

3. **Подсветка Enter при готовности** -- когда budgetInputValue не пустой, кнопка Enter слегка пульсирует (alpha 0.8 <-> 1.0, duration 800ms). Подсказка: "Нажмите для продолжения".

4. **Появление UI после завершения** -- все элементы появляются одновременно с DecelerateInterpolator за 400ms. Создает ощущение "приложение ожило".

5. **Haptic feedback** -- на Enter при сохранении бюджета и при выборе даты: `view.performHapticFeedback(HapticFeedbackConstants.CONFIRM)` (API 30+, совместимо с minSdk 29).

## 8. Метрики успеха (что измерять после реализации)

1. **Setup completion rate** -- % пользователей, завершивших оба шага (бюджет + дата). Цель: >90%.
2. **Time to first spend** -- время от первого запуска до первой записанной траты. Цель: <30 секунд.
3. **Day-1 retention** -- % пользователей, вернувшихся на следующий день. Базовая линия: измерить до и после.
4. **Setup abandon step** -- на каком шаге пользователи бросают (1 = бюджет, 2 = дата). Цель: <5% abandon на каждом шаге.
